# I18n

## Objectif

Préparer la traduction en français puis dans d'autres langues sans changer le moteur de jeu.

La V1 narrative utilise `en` comme fallback complet. Le livre 1 complet (`ch1` à `ch11b`), ses achievements, et les stats sont aussi disponibles en `fr`. Le shell UI est disponible en `en` et `fr`.

Le modèle est séparé :

- graphe logique ;
- messages narratifs affichés ;
- messages UI.

## Principe

La logique ne doit jamais être traduite.

Ne pas traduire ou modifier :

- scène IDs ;
- targets ;
- variables ;
- conditions ;
- assignments ;
- special values ;
- achievement variables ;
- ordre logique des blocs.

Traduire seulement :

- paragraphes ;
- choix ;
- titres/captions d'achievements ;
- libellés de stats ;
- UI shell.

## Format Actuel

Locale anglaise :

```text
content/canonical/v1/locales/en/<chapterId>.json
content/canonical/v1/locales/en/achievements.json
content/canonical/v1/locales/en/stats.json
content/canonical/v1/locales/en/ui.json
content/canonical/v1/locales/fr/ch1.json
content/canonical/v1/locales/fr/ch2.json
content/canonical/v1/locales/fr/ch3.json
content/canonical/v1/locales/fr/ch4.json
content/canonical/v1/locales/fr/ch5.json
...
content/canonical/v1/locales/fr/ch11b.json
content/canonical/v1/locales/fr/achievements.json
content/canonical/v1/locales/fr/stats.json
content/canonical/v1/locales/fr/ui.json
```

Forme :

```json
{
  "locale": "en",
  "chapterId": "ch1",
  "messages": {
    "ch1.Ch1_Intro1.p1": "They say there is...",
    "ch1.Ch1_Intro1.c1": "Excited"
  }
}
```

Le graphe référence uniquement les `messageId`.

## Sources Éditables

Les fichiers UI viennent de :

```text
content/ui-locales/en.json
content/ui-locales/fr.json
```

Ils sont packs en runtime sous les clés `locales/en/ui` et `locales/fr/ui`. `index.uiLocales` liste les locales UI disponibles.

Les fichiers narratifs traduits viennent de :

```text
content/story-locales/en/stats.json
content/story-locales/fr/ch1.json
content/story-locales/fr/ch2.json
content/story-locales/fr/ch3.json
content/story-locales/fr/ch4.json
content/story-locales/fr/ch5.json
...
content/story-locales/fr/ch11b.json
content/story-locales/fr/achievements.json
content/story-locales/fr/stats.json
```

Ils sont copiés en canonique puis packés sous les clés `locales/<locale>/<bundle>`. `index.storyLocales` liste les langues de récit disponibles.

## Langue Globale Et Fallback

Le setting de langue FR/EN est global :

- `settings.uiLocale` pilote le shell d'interface ;
- `settings.locale` et `GameState.locale` pilotent le récit, les achievements et les stats ;
- changer de langue recharge les packs texte, mais ne reset pas la scène, les variables, les achievements, l'historique ou le `historyDigest`.

Le fallback runtime reste `en`. Si un chapitre n'existe pas dans la locale choisie, il s'affiche en anglais, tandis que l'interface, les stats et les achievements traduits disponibles restent en français.

## Ajouter Une Langue

Future procédure attendue :

1. générer un squelette de locale depuis `en` ;
2. conserver exactement les mêmes `messageId` ;
3. remplir les valeurs traduites ;
4. valider la couverture ;
5. générer les runtime packs de la locale.

Chemin source cible :

```text
content/story-locales/fr/<chapterId>.json
```

Chemin canonique généré :

```text
content/canonical/v1/locales/fr/<chapterId>.json
```

Puis pack runtime :

```text
src/generated/packs/locales__fr__<chapterId>.ts
```

## Fallback

Le runtime tombe sur `en` si un pack locale n'existe pas.

Pour une vraie locale future, ajouter une validation stricte :

- toutes les clés `en` doivent exister dans la locale ;
- pas de clés en plus sauf convention explicite ;
- rapport de couverture par chapitre ;
- stats traduites avec les mêmes clés que `en/stats.json` ;
- achievements traduits pour chaque chapitre couvert par la locale, avec clés `title` et `caption` valides par rapport à `en` ;
- test que le changement de langue conserve l'état joueur.

## Export Traduction

Format d'échange recommandé pour la suite :

- JSON plat pour itération rapide ;
- XLIFF pour workflow traduction plus professionnel.

Le format runtime doit rester les packs générés, pas XLIFF.

## Contributions Publiques

La PWA peut afficher un formulaire de correction sur un paragraphe ou choix visible. Pour un texte narratif, le formulaire cible le paragraphe affiché dans le `messageId` via `segmentIndex`; il ne sert pas à réécrire tout un bloc multi-paragraphes. Ce formulaire ne modifie pas les fichiers de traduction et ne crée pas de PR directement.

Flux attendu :

1. le lecteur propose une correction anonyme par défaut ;
2. le payload envoie les IDs techniques cachés et les hash du texte courant/source, limités au segment pour une correction de paragraphe ;
3. `services/translation-api` stocke la proposition ;
4. un mainteneur accepte, rejette ou édite une version finale ;
5. les propositions acceptées sont groupées dans un changeset ;
6. GitHub Actions applique le changeset dans `content/story-locales/<locale>/<chapter>.json`, en remplaçant seulement le segment cible pour une correction de paragraphe ;
7. `pnpm content:all`, `pnpm check`, `pnpm test` et `pnpm build` doivent passer avant la PR.

Une proposition obsolétisée par une correction déjà mergée est détectée par `currentTextHash` et doit être marquée `stale`, pas forcée. Pour un paragraphe, seul le hash du segment cible est comparé : un changement dans un paragraphe voisin du même `messageId` ne suffit pas à rendre la proposition obsolète.

Le pseudo de crédit est facultatif et modérable. L'email de suivi est facultatif, confirmé par lien, et l'accord peut être réutilisé pendant un an dans le même navigateur via un jeton local. L'API conserve seulement une preuve HMAC/hash du consentement ; l'email brut reste limité aux contacts de propositions en attente, puis est supprimé après refus, stale ou publication.

## UI Shell

Les libellés UI sont dans des bundles source :

```text
content/ui-locales/en.json
content/ui-locales/fr.json
```

Ils sont générés puis chargés comme packs runtime :

- `locales/<locale>/ui`
- fallback `locales/en/ui`

Cela concerne :

- Saves ;
- Stats ;
- résultats de stat checks après choix ;
- Achievements ;
- Settings ;
- About ;
- notices d'erreur ;
- labels export/import.

`content:validate` vérifie que les locales UI ont exactement les mêmes clés que `en` et que les packs correspondants existent.
